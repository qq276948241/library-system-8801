# 青檀图书馆借阅管理系统 - 代码理解文档

## 一、项目总览

本项目是一个基于前后端分离架构的学校图书馆借阅管理系统：

| 层级 | 技术栈 | 说明 |
|------|--------|------|
| 前端 | Vue 3 + TypeScript + Vite + Tailwind CSS | 单页应用，含前台3页+后台3页 |
| 后端 | Go + Gin + SQLite(sqlx) + JWT | RESTful API，端口默认8080 |
| 数据库 | SQLite (WAL模式) | 文件库 `backend/library.db` |

## 二、后端模块详解

后端代码位于 `backend/` 目录，共 11 个 Go 源文件，按职责划分为以下模块：

### 2.1 入口与路由 - [main.go](file:///d:/code/ai-prompt/solo-20/repos/repo42/project42/backend/main.go)

**职责**：程序入口、数据库初始化、路由注册、中间件配置。

│  ├─ PUT  /auth/password    修改密码

│  ├─ GET  /books/categories 分类统计
│  ├─ GET  /books/:id        图书详情
│  ├─ GET  /borrows          借阅记录列表
│  ├─ POST /borrows          借书
│  └─ POST /borrows/:id/return  还书

└─ admin 路由组（需 AdminRequired 中间件，基于auth组叠加）
   ├─ POST   /books          新增图书
   ├─ PUT    /books/:id      修改图书

   ├─ GET    /users          用户列表

**中间件层级**：`admin` 路由组嵌套在 `auth` 路由组内，即管理员接口需同时通过 JWT 认证 + 角色校验。

### 2.5 借阅模块 - [borrows.go](file:///d:/code/ai-prompt/solo-20/repos/repo42/project42/backend/borrows.go)

**职责**：借阅记录查询（JOIN 优化）、借书、还书。

#### 2.5.1 查询优化（消除 N+1）

**原问题**：先查 `borrow_records`，再对每条记录分别查 `books` 和 `users`，3N+1 次查询。

**现方案**：定义 `borrowJoinRow` 结构体 + LEFT JOIN 一次性查询，所有字段用 `*指针` 承载 LEFT JOIN 的 NULL 值。

```go
// borrowJoinSQL 核心结构
SELECT br.*,
       b.id AS b_id, b.title AS b_title, ...    -- 图书字段（左连接，可能NULL）
       u.id AS u_id, u.username AS u_username, ... -- 用户字段（左连接，可能NULL）
FROM borrow_records br
LEFT JOIN books b ON br.book_id = b.id
LEFT JOIN users u ON br.user_id = u.id
```

`toBorrow()` 方法将 `borrowJoinRow` 转为 `Borrow`，非NULL的指针字段挂载到 `Borrow.Book` / `Borrow.User`。查询从 3N+1 次降为 1 次。

#### 2.5.2 借书流程（并发安全）

`handleBorrow` 的**原子更新**设计：

```
步骤1：开启事务 tx := db.MustBegin()
步骤2：INSERT 借阅记录（借期=当前日期+30天）
步骤3：条件UPDATE扣减库存
        UPDATE books SET available_copies = available_copies - 1
        WHERE id=? AND available_copies > 0
步骤4：检查 RowsAffected
        - = 0：库存不足，ROLLBACK，返回 400
        - = 1：COMMIT，返回成功
```

**为什么能防并发**：SQLite 的 UPDATE 是原子操作（单条语句级），`AND available_copies > 0` 把"检查-扣减"合并成一个原子步骤。即使两个请求同时到达，只有一个能匹配条件并获得 `RowsAffected=1`，另一个得 0 被回滚。

#### 2.5.3 还书流程

`handleReturn` 核心逻辑：

```
步骤1：查借阅记录，校验存在性 + 权限（本人或管理员）+ 未归还
步骤2：UPDATE borrow_records SET return_date=now, status='returned'
步骤3：UPDATE books SET available_copies = available_copies + 1  -- 直接+1，不用MIN兜底
步骤4：提交事务
```

**为什么移除 MIN 兜底**：原代码 `MIN(available+1, total)` 在管理员改小 `total_copies` 时会导致归还不增加库存。现改为依赖 `handleUpdateBook` 的前置校验（`total >= 已借出数`），保证任何时候 `已借出数 ≤ total`，因此归还后 `available+1 ≤ total`，无需 MIN。

## 三、借书还书完整流程

### 3.1 借书流程（含并发保护）

```
 前端 [借书按钮]
     │  POST /api/borrows  { book_id: 12 }
     │  Header: Authorization: Bearer <JWT>
     ▼
 Gin 路由层
     │
     ├─► AuthRequired 中间件
     │     ├─ 解析 JWT → 注入 uid=2 (zhangsan)
     │     └─ 失败返回 401
     │
     ▼ handleBorrow
     │
     ├─ 1. 解析 JSON 参数（book_id 必填）
     │
     ├─ 2. 开启事务  tx := db.MustBegin()
     │
     ├─ 3. INSERT 借阅记录
     │     借期默认 +30 天（SQLite datetime('now','+30 days')）
     │
     ├─ 4. 条件 UPDATE 扣减库存 ⭐原子操作
     │     UPDATE books
     │       SET available_copies = available_copies - 1
     │     WHERE id = 12 AND available_copies > 0
     │
     ├─ 5. 检查 RowsAffected
     │     │
     │     ├─ = 0 → tx.Rollback() → 返回 400"该书暂无可借副本"
     │     │
     │     └─ = 1 → tx.Commit() → 返回 200
     │
     ▼
 前端收到响应
     ├─ 成功：Toast 提示 + 刷新借阅列表/图书库存
     └─ 失败：Toast 错误消息
```

**并发场景说明**：假设库存=1，张三和李四同时点击借书：
- 张三的 UPDATE 先执行：匹配 `available_copies > 0`，扣减为 0，RowsAffected=1 → 成功
- 李四的 UPDATE 后执行：`available_copies` 已为 0，不匹配 WHERE 条件，RowsAffected=0 → 回滚，返回"暂无可借副本"

### 3.2 还书流程

```
 前端 [归还按钮]
     │  POST /api/borrows/18/return
     │  Header: Authorization: Bearer <JWT>
     ▼
 Gin 路由层
     │
     ├─► AuthRequired → 注入 uid/role
     │
     ▼ handleReturn
     │
     ├─ 1. 事务内查询借阅记录
     │     SELECT id, user_id, book_id, return_date
     │     FROM borrow_records WHERE id=18
     │
     ├─ 2. 校验
     │     ├─ 记录不存在 → 404
     │     ├─ 非本人且非管理员 → 403
     │     └─ 已归还(return_date NOT NULL) → 400
     │
     ├─ 3. UPDATE 借阅记录
     │     SET return_date = now, status = 'returned'
     │
     ├─ 4. UPDATE 图书库存
     │     SET available_copies = available_copies + 1
     │     (直接+1，配合 handleUpdateBook 的校验保证不越界)
     │
     ├─ 5. Commit → 返回 200
     │
     ▼
 前端收到响应 → Toast 提示 + 刷新列表
```

### 3.3 管理员修改馆藏流程（数据一致性保证）

```
 后台 [保存图书]
     │  PUT /api/books/14  { ..., total_copies: 3 }
     ▼
 handleUpdateBook
     │
     ├─ 1. 查询旧数据 SELECT total, available
     │     old.Total=5, old.Avail=4  → 已借出 = 5-4 = 1
     │
     ├─ 2. 校验 in.TotalCopies(3) >= borrowed(1) ✅
     │     （若 < 则拒绝，返回 400）
     │
     ├─ 3. 计算 newAvail
     │     newAvail = old.Avail + (new.Total - old.Total)
     │              = 4 + (3 - 5) = 2
     │     夹取边界：0 <= 2 <= 3 ✅
     │
     └─ 4. UPDATE 同时写入 total=3, available=2
```

**归还后的预期**：`available_copies(2) + 1 = 3 = total_copies`，归还后刚好填满。

### 4.2 路由与权限（[router/index.ts](file:///d:/code/ai-prompt/solo-20/repos/repo42/project42/src/router/index.ts)）

| 路由层级 | 路径 | 页面 | 守卫 | 使用角色 |
|---------|------|------|------|---------|
| 根 | `/login` | LoginPage | - | 所有人 |
| 前台组 | `/` (FrontLayout) | | `requireAuth` | 登录用户 |

| | `/borrows` | BorrowsPage | | 学生/管理员 |
| | `/profile` | ProfilePage | | 学生/管理员 |

| | `/admin/books` | AdminBooksPage | | 管理员 |
| | `/admin/users` | AdminUsersPage | | 管理员 |

**守卫逻辑**：
- `requireAuth`：未登录跳 `/login?redirect=原路径`
- `requireAdmin`：未登录跳登录，已登录但非管理员强制跳前台 `/books`

**状态恢复**：页面刷新时 `ensureReady()` 调用 `useAuth().restore()`，用 localStorage 中的 Token 请求 `/api/auth/me` 恢复用户对象，保证刷新后路由守卫判断正确。

### 4.4 核心组合式函数

#### [useAuth.ts](file:///d:/code/ai-prompt/solo-20/repos/repo42/project42/src/composables/useAuth.ts) - 登录态管理

- `state.user`：全局响应式用户对象（reactive，只读暴露）
- `restore()`：用 localStorage 中的 token 请求 `/api/auth/me`，避免刷新丢失登录态（防抖：多次调用共享同一个 Promise）
- `login()`：请求登录 → 存 token → 写 state
- `logout()`：清空 state + localStorage
- 401 全局钩子：API 层收到 `code=401` 时自动登出

#### [api.ts](file:///d:/code/ai-prompt/solo-20/repos/repo42/project42/src/lib/api.ts) - API 客户端

- 统一 `request()` 封装：自动加 `Authorization: Bearer`、自动解析 `{code, msg, data}`、`code !== 0` 抛 `ApiError`
- 按业务分组：`authApi` / `booksApi` / `borrowsApi` / `usersApi` / `statsApi`
- 完整 TypeScript 类型定义（与后端模型一一对应）

#### [format.ts](file:///d:/code/ai-prompt/solo-20/repos/repo42/project42/src/lib/format.ts) - 到期提醒核心

| 导出 | 作用 |
|------|------|
| `daysUntilDue(dueDate, returnDate)` | 计算距到期天数（负数=逾期，null=已归还或无效） |
| `type DueLevel = 'normal' \| 'warning' \| 'danger' \| 'returned'` | 到期警告级别 |
| `getDueLevel(dueDate, returnDate)` | 按天数映射级别（<0→逾期，≤3→警告，其余→正常） |
| `getDueStyles(level)` | 级别→样式类（边框/背景/文字/徽章 4 种 Tailwind class） |

所有页面的到期提醒视觉效果都基于此工具函数，保证视觉一致性。
